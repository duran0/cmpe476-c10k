1. Overview
In Chapter 2 of the course we discussed the C10K problem: the challenge of designing a single
server that can sustain ten thousand (and eventually millions of) simultaneous client connections.
We contrasted the simple thread-per-connection model — which does not scale past a few
thousand clients because every thread consumes roughly a megabyte of stack and forces the
kernel to context-switch among thousands of runnable threads — with the event-loop model used
by production systems such as Nginx, Redis, HAProxy, and Node.js, which multiplexes many
connections on a single thread using non-blocking I/O and a readiness notification mechanism
(epoll on Linux, kqueue on BSD, IOCP on Windows).
In this project you will build both architectures yourselves, from the ground up, in C. You will then
run them side by side under load and write a short report comparing their memory footprint,
throughput, and latency. The goal is not merely to produce working code: it is to turn the abstract
numbers we drew on the slides (1 MB per thread, thousands vs. millions of connections) into
measurements you have produced on your own laptop, so the trade-offs become something you
have felt rather than something you have read.

4. Deliverables
You must submit the following, packaged as described in Section 11:

Part A — threadserv (30 points)
A thread-per-connection TCP server. A new POSIX thread (pthread_create) is spawned for each
accepted client. Each worker thread reads lines from its socket, processes them through the
shared protocol functions, writes the response back, and terminates when the client closes the
connection or sends QUIT. The main thread loops on accept(). Global state (connection counts)
must be protected by a mutex.

Part B — epollserv (40 points)
A single-threaded event-loop TCP server using epoll(7) in edge-triggered mode. All sockets
(including the listening socket) must be non-blocking. The server must correctly drain each ready
socket until read() returns EAGAIN, accumulate partial lines in a per-connection buffer, and
process each complete line through the shared protocol functions. No pthreads, no fork, no busy-
waiting.

5. Wire Protocol
Both servers speak the same text-based protocol. One request per line, one response per line,
all lines terminated by '\n'. The server tolerates an optional '\r' before the '\n' (so telnet and netcat
both work). The command word is case-sensitive and uppercase; there is no leading whitespace
before the command.

 Request                    Response                          Meaning

 PING                       PONG                              Liveness check.

 ECHO <text>                <text>                            Echo everything after the single
                                                              separating space.

 ECHO                       (empty line)                      ECHO with no payload returns an
                                                              empty response line.

 TIME                       <unix_seconds>                    Current wall-clock time in seconds
                                                              since the UNIX epoch.

 STATS                      <active_connections>              Current number of established client
                                                              connections.

 QUIT                       BYE                               Server sends BYE then closes the
                                                              socket.
 anything else              ERR unknown_command               Unknown command; connection stays
                                                              open.
 line > 1024 bytes (no      ERR line_too_long                 Server emits the error line, then closes
 '\n')                                                        the socket.

  Note

  Responses are at most 1088 bytes. These constants are defined as MAX_LINE_LEN and
  MAX_RESPONSE_LEN in server_api.h — use them, do not hard-code the numbers.

6. Fixed API — server_api.h
You receive a header file, server_api.h, that defines the types and function signatures your
implementation must expose. Do not modify this file. The CUnit test harness #includes it and links
against your object files; any change to the header (renaming a field, changing a signature, adding
arguments) will cause link-time failure and you will lose all automated-test points.
The following five functions are graded by CUnit. You must provide exactly these signatures, with
exactly this behaviour.

6.1 parse_request
  int parse_request(const char *line, request_t *out);

Parses one line (already stripped of its trailing newline) into the output struct. The command word
is matched exactly: PING, ECHO, TIME, STATS, QUIT. Any leading whitespace, lowercase, or
unrecognised token must produce CMD_UNKNOWN. For ECHO, exactly one space separates
the command from the payload; any further spaces belong to the payload (so "ECHO hi" has arg
== " hi", with a leading space). Returns 0 on success (including when the command is
CMD_UNKNOWN), or -1 if either argument is NULL.

6.2 format_response
  int format_response(const request_t *req,
             const server_state_t *st,
             char *out, size_t outlen);

Writes the response line (without the trailing newline) into out, truncating to outlen-1 bytes and
always NUL-terminating. Returns the number of bytes written (excluding the NUL) on success, or
-1 on error. The mapping from request kind to response text is given by the table in Section 5.
The caller is responsible for appending '\n' before writing to the socket.

6.3 set_nonblocking
  int set_nonblocking(int fd);

Sets the O_NONBLOCK flag on fd using fcntl(2). Returns 0 on success, -1 on failure (with errno
set). This is required by epollserv for the listening socket and every accepted socket.

6.4 buffer_append
  int buffer_append(conn_buf_t *b, const char *data, size_t n);

Appends n bytes of data to the per-connection line buffer. If appending would overflow
CONN_BUF_CAPACITY and no '\n' is present in the combined content, the buffer is reset and
the line_too_long flag is latched to 1; subsequent take-line calls must surface this exactly once
(see next function). Returns 0 on success, -1 on NULL input.

6.5 buffer_take_line
  int buffer_take_line(conn_buf_t *b, char *out, size_t outmax);

Extracts the next complete line (up to but not including '\n', with any '\r' before the '\n' also stripped)
into out, removes it from the buffer, and returns 1. Returns 0 if the buffer does not yet contain a
complete line. If the line_too_long flag is set, returns 1 exactly once with out[0] == '\0' and then
clears the flag — this is how the epoll server learns that the peer has sent an over-length line and
should be disconnected. Returns -1 on NULL input.

  ⚠ Important
  Because the CUnit harness links against these exact symbols, you should place
  parse_request and format_response in a file called protocol.c, and set_nonblocking,
  buffer_init, buffer_append, buffer_take_line in a file called buffer.c. The Makefile assumes
  these file names. You may add static helper functions inside either file; keep all non-static
  externally visible symbols to what server_api.h declares.



7. Part A — threadserv
The entry point is main in threadserv.c. Command-line argument: the TCP port to listen on (default
9090 if no argument is given).
Required structure:
    1. The main thread creates an AF_INET / SOCK_STREAM socket, sets
       SO_REUSEADDR, binds it to INADDR_ANY on the given port, calls listen(), then loops
       on accept().
    2. For every accepted client, main spawns a detached POSIX thread (pthread_create
       followed by pthread_detach) that runs a worker function. The worker receives the client
       file descriptor as its argument.
    3. Inside the worker: increment active_connections (under the mutex), then loop — read()
       from the socket, append into a conn_buf_t, extract complete lines with buffer_take_line,
       call parse_request and format_response, write the response followed by '\n' back to the
       socket. Exit the loop on EOF, on QUIT, on line_too_long (after sending ERR
       line_too_long), or on a write error.
    4. Before the worker returns, close the socket and decrement active_connections (under
       the mutex).
    5. Install a SIGPIPE handler — or use SIG_IGN — so that a client closing the socket mid-
       write does not kill the server. Check the return value of write() and treat failures as a
       signal to end the connection.
Required properties that the functional test will check:

   •   STATS returns a correct, monotonically plausible count (no double counting, no leaks
       after clients disconnect).
   •   The server does not segfault, leak file descriptors, or exit when clients abruptly
       disconnect.
   •   The server handles at least 500 concurrent connections without crashing.


8. Part B — epollserv
The entry point is main in epollserv.c. Command-line argument: the TCP port (default 9091).
Required structure:
   1. Create and bind the listening socket as in Part A. Call set_nonblocking on it. Create an
      epoll instance with epoll_create1, and register the listening socket for EPOLLIN.
   2. Run an event loop around epoll_wait. When the listening socket is readable, loop calling
      accept() until it returns -1 with EAGAIN (accept-all-you-can). Set every accepted socket
      non-blocking and register it with EPOLLIN | EPOLLET, attaching a per-connection struct
      (containing at least the fd and a conn_buf_t) to the event's data.ptr.
   3. When a client socket is reported readable, loop calling read() until it returns -1 with
      EAGAIN (required by edge-triggered mode: you only get one notification per readiness
      transition, so you must drain). Feed each chunk into buffer_append. Then loop calling
      buffer_take_line until it returns 0, processing each line.Define constant INTEST as 1.




   4. On QUIT, on line_too_long, on read()==0 (peer closed), or on any unrecoverable error,
      remove the fd from epoll, close it, free the per-connection struct, and decrement
      active_connections.
   5. The entire server runs on one thread. No pthreads, no fork.

  ⚠ Important
  Edge-triggered mode is the subtle part. If you forget the drain loop, or if you use EPOLLIN
  without EPOLLET and get level-triggered semantics by accident, the test harness will still
  work for a few clients but will deadlock under the 10,000-connection benchmark — which is
  precisely the point of choosing edge-triggered in the first place.

Required properties that the functional test will check:
   •   Works correctly when many clients send partial lines (one byte at a time).
   •   Works correctly when one client sends many lines back-to-back in a single TCP
       segment.
   •   Handles at least 10,000 concurrent connections without crashing. (The exact threshold
       depends on your ulimit -n; raising it to 65,535 is part of the project, not an accident.)

9. Part C — Benchmark Report
Produce a short report (2–3 pages, PDF) that compares threadserv and epollserv under identical
load. The supplied client generator client_flood.c opens N concurrent TCP connections, sends M
requests on each, collects wall-clock latency, and prints aggregate statistics. You are free to
modify it or replace it, as long as your methodology is reproducible. Note that each group develop
and test their project under different circumstances; so, I need to know your configuration to
evaluate your submission.
Your report must contain, at minimum:
   •    a description of your test setup (machine, kernel version, ulimit -n, any sysctl tuning);
   •    a table of memory (RSS -- Resident Set Size, via /proc/<pid>/status) and throughput
        (requests per second) for each server at 1,000, 5,000, and 10,000 concurrent clients;
   •    at least one chart — either a bar chart comparing RSS or a line chart of throughput vs.
        concurrency;
   •    a one-paragraph discussion interpreting the results in terms of the thread-stack
        overhead and context-switching cost that we discussed in Chapter 2;
   •    one honest limitation of your methodology (e.g., "localhost loopback hides the effect of
        network latency").

  Note
  You are not expected to beat Nginx. Typical results on a laptop: threadserv peaks around
  2,000–5,000 concurrent clients before RSS climbs past several gigabytes; epollserv handles
  10,000 clients with a few tens of megabytes of RSS. If your numbers differ wildly, include that
  in your discussion rather than hiding it — honest negative results are graded favourably.
